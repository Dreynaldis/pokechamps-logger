# ADR 003: Backend -- Go + Chi over NestJS

**Status:** Accepted   ·   **Date:** 2026-06-27

## Context

The backend is a REST API serving CRUD endpoints for teams and matches, auth endpoints (email/password + OAuth), and read-only reference data endpoints with trigram autocomplete.
There is no complex real-time requirement, no heavy computation, and no existing codebase to extend.
The developer has existing experience with Node.js, Express, and TypeScript (including NestJS), and is open to learning a new language.

## Decision

Use Go with the Chi HTTP router and GORM.
Use `golang-jwt/jwt` for token signing, `markbates/goth` for OAuth provider adapters, and `go-playground/validator/v10` for request body validation.
Bridge the type gap between Go and the TypeScript frontend via an OpenAPI spec generated by `swaggo/swag`, consumed by `openapi-typescript` to generate TypeScript types.

## Options Considered

### Option A -- Go + Chi
- Pros: single static binary deployment (no runtime version to manage on the server); explicit, no-magic routing and middleware -- auth guard is a plain middleware function applied to a route group, visible at the route definition; Go's goroutine model handles concurrent writes efficiently; strong standard library; genuinely different paradigm from Node.js, making it a high learning-value choice; lower memory footprint per request than Node.js.
- Cons: no shared type system with the TypeScript frontend -- requires OpenAPI codegen bridge; smaller ecosystem of auth-specific plugins compared to NestJS/Passport; Go's error-handling verbosity (explicit `if err != nil`) is unfamiliar at first; the developer will be learning Go idioms at the same time as building the app.

### Option B -- NestJS (TypeScript)
- Pros: TypeScript end-to-end means types can be shared directly with the SvelteKit frontend; DI, guards, and pipes are well-suited to the auth and validation patterns; the developer already has NestJS experience, so iteration speed would be higher initially.
- Cons: no meaningful learning gain (developer already knows this stack); NestJS's full framework machinery (modules, DI, decorators) is overkill for a simple CRUD API -- it adds ceremony without benefit at this scope; Node.js requires a runtime on the server; NestJS decorator-heavy patterns can obscure what is actually happening at the HTTP layer.

### Option C -- Rust + Axum
- Pros: best possible runtime performance; excellent type system; single binary deployment.
- Cons: ownership and lifetime system significantly steepens the learning curve; would slow Phase 0-3 substantially; overkill for this scale.

## Consequences

- **Positive:** simple, auditable middleware chain for auth; single binary deployment simplifies hosting; Go is a high-value addition to the developer's skillset; explicit error handling catches failure cases at compile time rather than at runtime.
- **Negative:** OpenAPI codegen is a required build step -- if the spec drifts from the implementation, type errors appear on the frontend; `goth` for OAuth has less documentation coverage than Passport.js strategies; initial development may be slower while Go idioms are being learned.

## API/contract impact

The OpenAPI spec (`openapi.json`) generated by `swaggo/swag` becomes the contract artifact between backend and frontend.
Any breaking API change must be reflected in the spec and regenerated types before the frontend can consume the new shape.
This is a discipline overhead that does not exist in a shared-TypeScript setup.
